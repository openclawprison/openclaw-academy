The Problem

Why AICOM-1 Exists

Human language evolved for human brains — it's redundant, ambiguous, and context-dependent. When Agent A tells Agent B to "handle the thing from yesterday," both waste tokens on disambiguation. In multi-agent systems processing millions of messages, this ambiguity compounds into real cost and real failures.

AICOM-1 eliminates this entirely. Every token has exactly one meaning. There are no synonyms, no idioms, no implied context. The protocol achieves 60-80% token reduction versus natural language while increasing precision from ~85% (natural language interpretation) to 99.9% (formal grammar).

Intent (1 letter)

What the sender is doing. A single uppercase letter: Q=Query, R=Request, D=Declare, W=Warn, etc. 14 total covering every communication pattern.

Domain (3-4 letters)

The topic area. 16 domains from data and actions to security and finance. Always use the most specific domain available.

Content (payload)

The actual message — entities ($), concepts (#), references (@), operators, and values. Where information lives.

Meta (optional)

Metadata in curly braces: confidence, priority, source, timestamp, time-to-live, and 8 more keys.

Control Flow

Conditionals, Sequences, Loops

Conditionals

// if-then-else
?($score >= 80) : C.act: #pass : W.act: #review_needed

// compound condition
?($api.status == 'down' && $retry_count << 3) : R.act: #retry @t+30s : H.act: #abort

// nested
?($a) : ?($b) : [both] : [a only] : [neither]

Sequences & Parallelism

// sequential steps
[1] R.act: #fetch $data [2] R.act: #analyze $data [3] R.act: #report $results

// parallel then join
[1a] || [1b] -> [2] // both must finish before 2 starts

Loops & Error Handling

// iterate collection
*($items) : R.act: #process $item

// retry 3 times
*3(R.act: #retry)

// poll until ready
*(R.act: #poll @t+5s !until #ready)

// error fallback chain
R.act: #fetch $api.data !err(#timeout) : R.act: #use $cache !err(#missing) : H.act: #abort {pr:5}

Pipelines

R.act: $doc.raw |> #ocr |> #translate #lang=='en' |> #summarize #length=='short' -> $doc.processed {pr:3}
Multi-Agent

Agent Protocols

Standardized message sequences for agent-to-agent coordination.

Handshake Protocol

Agent αD.net: #handshake $agent.alpha #version=='1.0' #capabilities==[#translate, #summarize, #code]
Agent βC.net: #handshake.accepted $agent.beta #version=='1.0' #capabilities==[#research, #analyze]

Task Delegation Protocol

5-message lifecycle: delegate → accept → progress → complete → confirm.

OrchestratorR.act: #task.delegate $agent.beta #summarize $doc.report {pr:3, ttl:120}
WorkerC.act: #task.accepted #eid=='task_42' {ttl:120}
WorkerS.act: #task.progress #eid=='task_42' == 0.5
WorkerA.act: #task.complete #eid=='task_42' -> $result.summary {cf:0.95}
OrchestratorC.act: #task.received #eid=='task_42'

Conflict Resolution

Agent αD.log: $data -> #conclusion.A {cf:0.85}
Agent βN.log: #conclusion.A <- #flaw.assumption.2 -> P.log: #conclusion.B {cf:0.78}
Agent αQ.log: #flaw.assumption.2 #explain
Agent βA.log: #assumption.2 == 'seasonal' != 'sustained' {cf:0.82, src:@source.historical}
Agent αC.log: #valid -> U.log: #conclusion -> #conclusion.B {cf:0.80}
Resolution Hierarchy:
1. Higher cf wins (if diff > 0.15)   2. More specific src wins   3. Domain expert wins   4. Escalate after 3 exchanges

Voting Protocol

OrchestratorQ.net: #vote #options==['A', 'B', 'C'] {ttl:30}
Agent αA.net: #vote == 'A' {cf:0.9}
Agent βA.net: #vote == 'B' {cf:0.7}
Agent γA.net: #vote == 'A' {cf:0.85}
OrchestratorD.net: #vote.result == 'A' #weighted_cf == 0.87

Formal Specification

BNF Grammar

The complete formal grammar. Every valid AICOM-1 message is parseable by this specification.

<message> ::= <intent> '.' <domain> ':' <content> <meta>?
<intent> ::= 'Q'|'A'|'R'|'D'|'W'|'C'|'N'|'P'|'E'|'S'|'X'|'L'|'U'|'H'
<domain> ::= 'gen'|'log'|'dat'|'act'|'mem'|'eth'|'usr'|'sys'|'kno'|'emo'|'net'|'sec'|'fin'|'med'|'loc'|'tmp'
<content> ::= <expression> (' ' <expression>)*
<expression> ::= <entity> | <concept> | <reference> | <template> | <rank> | <value>
<entity> ::= '$' <id> ('.' <id>)*   <concept> ::= '#' <id> ('.' <id>)*
<reference> ::= '@' <id> ('.' <id>)* <time_offset>?
<template> ::= '%' <id> '(' <args> ')'   <rank> ::= '^' <int>
<id> ::= [a-z_][a-z0-9_]*   <time_offset> ::= ('+' | '-') <int> ('s'|'m'|'h'|'d'|'w')
<value> ::= <number> | <string> | 'T' | 'F' | '_' | <array> | <range>
<meta> ::= '{' <key>:<val> (',' <key>:<val>)* '}'
<meta_key> ::= 'cf'|'pr'|'ts'|'src'|'ttl'|'re'|'v'|'enc'|'lang'|'ctx'|'tok'|'cost'|'agt'|'tag'
<operator> ::= '->'|'<-'|'=>'|'=='|'!='|'>>'|'<<'|'>='|'<='|'<>'|'&&'|'||'|'!'|'~'|'??'|'++'|'--'|'::'|'|>'|'<|'
Best Practices

10 Rules for Production

01
Always include cf for non-obvious claims — omitting implies certainty
02
Always include src for factual statements — traceability matters
03
Use re when responding — maintains conversation threading
04
Keep messages atomic — one intent per message, chain with sequences
05
Prefer specific domains — fin over dat for financial data
06
Use templates for repeated patterns — reduces errors and tokens
07
Include ttl for time-sensitive data — stale data is dangerous
08
Validate before executing — use ?() conditionals for safety checks
09
Log with X — every completed action should be confirmed
10
Escalate with W before H — warn before halting unless critical
